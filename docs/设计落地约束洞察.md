# 设计落地约束洞察

## 目标

本文件用于把“视觉设计 Skill”中的规范、规则、脚本、提示词和 command，前置转化为视觉洞察阶段的判断角度。

视觉洞察不直接生成设计稿，也不替代视觉设计 Skill 的组件调用流程。它要提前回答：

- 后续视觉设计要遵守哪些页面结构、组件、变量和状态约束。
- 当前需求中哪些 UI 决策会影响 MasterGo / 组件库落地。
- 哪些内容应进入视觉审查 Skill，作为设计产物验收基准。
- 哪些缺失材料会导致设计执行阶段无法稳定生成。

## 何时读取

当任务出现以下任一情况时，读取本文件：

- 用户提到输入法 APP、输入法APP-视觉设计、视觉设计 Skill、C2D、MasterGo、组件库、设计规范。
- 用户希望视觉洞察报告作为后续视觉设计或视觉审查的输入。
- PRD / 交互链路涉及输入法 APP 的一级页面、二级编辑页、键盘、底部弹层、弹窗、tabbar、支付/保存/生成链路。
- 用户提供视觉设计规范、组件库链接、规则、脚本、command 或提示词，希望纳入洞察依据。

## 使用边界

可以进入视觉洞察的内容：

- 页面结构约束
- 组件调用约束
- 变量与设计 token 约束
- 状态、空态、错误态、弹窗、键盘等必要 UI 覆盖
- 后续视觉审查可验证的检查点
- 设计执行阶段可能遇到的冲突和缺失风险

不要进入视觉洞察的内容：

- 具体 HTML / MasterGo 生成代码
- 组件 JSON 的完整 props 枚举
- 脚本实现细节
- 命令行安装步骤
- 纯工具执行说明

## 洞察角度

### 1. 设计落地可行性

洞察时要判断：

- 当前需求是否能由现有设计系统 / 组件库承接。
- 是否存在必须使用的核心组件。
- 是否有组件库缺失，需要后续用临时容器或补组件。
- 交互稿中的页面结构是否符合目标产品的固定界面结构。

可输出为：

| 当前需求页面 / 区域 | 落地约束 | 设计风险 | 洞察建议 |
|---|---|---|---|
| ... | 需要复用导航 / tab / keyboard / modal 等组件 | 组件缺失会导致设计稿不一致 | 设计前确认组件能力或补组件 |

### 2. 页面结构约束

输入法 APP 场景下，洞察阶段需要提前识别页面类型：

| 页面类型 | 洞察重点 |
|---|---|
| 一级频道页 | 是否需要顶部渐变、状态栏、一级导航、内容区和悬浮 tabbar |
| 二级 / 编辑页 | 是否需要二级导航、内容区、底部操作区、键盘或底部抽屉 |
| 弹窗 / 底部弹层 | 是否需要蒙层、弹层标题、内容列表、主按钮和取消入口 |
| 图片编辑 / 裁剪页 | 是否需要沉浸背景、裁剪区域、完成按钮、返回 / 关闭入口 |
| 键盘编辑页 | 是否必须保留真实键盘预览、可替换坑位、底部键盘安全区 |

视觉洞察要明确：当前需求属于哪种页面结构，后续视觉设计应优先保证哪些结构稳定。

### 3. 组件约束

洞察时不需要输出完整组件调用表，但要提前指出核心 UI 元素是否依赖组件。

输入法 APP / MasterGo 场景常见核心组件：

- 状态栏
- 一级界面顶部渐变
- `nav_bar_primary`
- `nav_bar_secondary`
- `tabbar`
- `keyboard`
- `button`
- `modal`
- tab / radio / checkbox / toggle
- card / list / placeholder

洞察判断方式：

- 如果关键路径必须出现某个组件，报告中要写成“设计落地约束”。
- 如果组件缺失会影响高保真交付，报告中要写成“风险 / 需补充”。
- 如果交互稿中有自定义区域但组件库可能没有，要提示使用临时容器并说明边界。

### 4. 变量与设计 token 约束

视觉洞察不输出具体变量值，但要提醒后续视觉设计：

- 主按钮、选中态、强调信息应优先使用设计系统高亮色。
- 正文、说明、标签、禁用态应遵守字体层级和文本色层级。
- 重要状态不能随意新建颜色，应使用语义色或已有变量。
- 紧凑卡片、tab、标签不应使用过大字号。
- 如果需求需要新颜色、新插画、新动效，要说明是否可能造成品牌一致性风险。

### 5. 状态覆盖约束

视觉洞察必须提前判断当前需求是否需要以下状态：

- 加载态
- 成功态
- 失败态
- 空状态
- 网络异常
- 权限不足
- 保存中 / 支付中
- 支付取消 / 支付失败
- 已购买 / 未购买
- 禁用态 / 不可点击原因

如果后续视觉审查需要验证这些状态，视觉洞察报告中要把它们写进 Checklist 或分支与边缘路径。

### 6. 视觉审查基准

当视觉洞察报告会作为视觉审查 Skill 的输入时，每条关键洞察应尽量可验收。

建议将洞察写成：

```text
设计关注点：
证据来源：
涉及路径：
涉及 UI 元素：
后续设计应体现为：
视觉审查时检查：
```

不要只写：

```text
优化视觉体验。
增强高级感。
提升吸引力。
```

### 7. 冲突与缺失风险

视觉洞察要提前标注以下风险：

- PRD 目标和交互截图冲突。
- visual-insight 洞察和视觉设计规范冲突。
- 组件库缺少关键组件。
- 交互截图文字无法识别，影响页面内容判断。
- 需求需要新组件，但没有设计系统沉淀。
- 一次生成多屏会导致组件调用漂移或细节不一致。

输出时放在“下一步需要补充”或“可执行设计建议”中。

## 输出要求

不要新增独立一级章节，除非用户明确要求。默认把设计落地约束合并到：

- 输入材料判断：是否提供设计规范、组件库、visual design skill 规则。
- 设计原则命中分析：标注“设计落地约束，非原则库直接推导”。
- 视觉洞察 Checklist：增加“组件可落地性 / 设计系统一致性 / 状态覆盖 / 审查基准”相关行。
- 可执行设计建议：将建议转成后续设计能执行、视觉审查能验证的动作。
- 下一步需要补充：列出缺失的组件库、规范、截图、状态页或审查基准。

## 示例

| 维度 | 当前判断 | 建议动作 | 当前需求需要重点关注什么 | 证据来源 | 优先级 |
|---|---|---|---|---|---|
| 组件可落地性 | 当前链路包含键盘预览和底部弹层，后续设计强依赖 keyboard 与弹层组件 | 设计前确认 keyboard、modal / drawer、button 组件能力，不足时标注临时容器边界 | 避免后续视觉设计自创组件导致和输入法 APP 规范割裂 | 交互截图 / 视觉设计规范 | 高 |
| 状态覆盖 | 保存、支付、上传、网络异常都会影响闭环 | 在洞察中明确加载、失败、取消、成功态，作为视觉审查基准 | 让视觉审查能验证设计稿是否覆盖完整闭环 | PRD / 交互文档 | 高 |
