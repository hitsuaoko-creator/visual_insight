# 视觉洞察

## 作用

这是 Codex 的 Skill 入口，不是唯一知识库。真正的通用知识源在 `docs/` 和 `prompts/`。

本 Skill 的任务是让 Codex 在用户提出视觉洞察相关需求时，自动读取通用知识库，并按统一工作流输出视觉洞察。

## 必读文件

处理视觉洞察任务前，必须读取：

1. `docs/视觉洞察工作流.md`
2. `docs/视觉原则知识库.md`

如果任务包含竞品分析，读取：

3. `docs/竞品UXUI分析框架.md`

如果用户需要结构化交付物，读取：

4. `docs/输出模板.md`

如果用户需要可复制 Prompt，读取：

5. `prompts/视觉洞察主Prompt.md`

如果任务涉及输入法 APP、MasterGo、组件库、视觉设计 Skill、设计规范、C2D，或视觉洞察报告将作为视觉审查 Skill 的输入，读取：

6. `docs/设计落地约束洞察.md`

## 引导式交互

当用户没有一次性提供完整材料时，主动引导，但不要卡住。

优先询问：

1. 这个需求的 PRD 或需求背景是什么？
2. 关键路径是什么？
3. 有没有当前链路截图、旧版 UI 截图或交互稿？
4. 是否需要包含竞品 UXUI 分析？

如果用户没有竞品材料，输出竞品采集清单，而不是编造竞品结论。

如果用户没有品牌规范，询问产品希望被感知为何种品牌气质，例如：专业、年轻、可信、高效、温暖、科技、娱乐、高端、安全、创作感。

## 强制知识库巡视 Hook

当用户提出输入法APP-视觉洞察、视觉洞察、UI 视觉分析、竞品 UXUI 分析、基于 PRD 的视觉设计洞察、设计前置 Checklist、UI 方案前设计判断、视觉走查或设计评审前分析时，必须先执行“视觉洞察知识库巡视流程”，再输出最终内容。

不要仅凭通用审美、个人偏好或泛泛设计经验直接给结论。

执行要求：

- 先读取 `docs/视觉洞察工作流.md` 和 `docs/视觉原则知识库.md`。
- 根据任务需要读取竞品分析框架、输出模板和主 Prompt。
- 根据任务需要读取设计落地约束洞察，把视觉设计 Skill 的规范、规则、脚本、提示词和 command 转化为设计落地约束，不要照搬执行步骤。
- 建立“业务目标 - 路径步骤 - UI 元素 - 设计原则 - 证据 - 指标”的映射。
- 每条重要建议尽量包含问题/机会点、证据来源、路径步骤、相关 UI 元素、UI 设计动作、命中原则、预期影响、验证指标和优先级。
- 如果视觉洞察报告会作为视觉审查输入，每条高优先级建议尽量补充“后续视觉审查检查点”。
- 如果某条建议来自业务判断，而不是原则库直接推导，明确标注。
- 如果某条建议来自设计规范、组件库或视觉设计 Skill，明确标注“设计落地约束，不属于原则库直接推导”。

## 禁止事项

- 不编造竞品截图中不存在的细节。
- 不把设计原则当百科解释。
- 不把“视觉风格选择库”纳入分析，除非用户再次明确要求。
- 不默认所有需求都需要营销化视觉。
- 不用审美偏好替代业务路径判断。
- 不只写“提升高级感”“优化体验”“更清晰”“更年轻化”等空话，除非转成具体 UI 动作。

## 输出

优先按 `docs/输出模板.md` 输出。必要时可压缩，但至少保留：

- 输入材料判断
- 需求视觉目标
- 设计原则命中分析
- 关键视觉路径
- 竞品 UXUI 洞察或竞品采集清单
- 视觉洞察 Checklist
- 可执行设计建议
