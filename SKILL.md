# 视觉洞察

## 目标

把用户提供的 PRD、需求背景、角色、链路截图、竞品截图、品牌规范和历史 UI 问题，转成一份可进入 UI 方案设计与设计评审的视觉洞察报告。

本 Skill 不负责直接绘制 UI，不输出泛泛审美建议，不纳入视觉风格选择库。它的核心是：用视觉原则知识库作为判断依据，把每条视觉建议绑定到业务路径、UI 元素、用户判断和验证指标。

## 引导式交互

当用户没有一次性提供完整材料时，主动进入引导式交互。不要卡住，也不要要求用户必须补齐所有材料后才开始。

优先询问 4 个关键问题：

1. 这个需求的 PRD 或需求背景是什么？
2. 你希望分析的是哪个关键路径？
3. 有没有当前链路截图、旧版 UI 截图或交互稿？
4. 是否需要包含竞品 UXUI 分析？

如果用户需要竞品分析但没有提供竞品材料，继续引导：

- 请提供竞品名称、竞品截图、竞品链接中的任一种。
- 如果都没有，请让用户提供业务类型，并输出“竞品采集清单”和“应采集页面”。

如果用户没有提供品牌规范，询问产品希望被感知为何种品牌气质：

- 专业、年轻、可信、高效、温暖、科技、娱乐、高端、安全、创作感。

如果用户只提供一句话需求，先输出：

- 已有信息
- 缺失信息
- 基于现有信息可先判断的内容
- 建议补充材料清单
- 初步视觉洞察框架

## 推荐输入格式

用户可以按以下结构提供材料，但不要强制要求完全一致：

```text
使用 visual-insight skill。

需求名称：
【粘贴】

需求背景：
【粘贴】

PRD：
【粘贴正文，或上传文档/截图】

目标角色：
【如没有，请基于 PRD 反推】

关键路径：
【例如：首页 → 点击创作 → 进入创作页 → 选择模板 → 生成内容 → 保存/发布】

交互链路截图：
【粘贴截图，或说明稍后上传】

竞品材料：
【竞品名称 + 截图/链接；没有则写：暂无，需要你告诉我该看哪些竞品页面】

品牌规范：
【品牌关键词、设计系统、色彩、字体、组件规范；没有则写：暂无】

历史 UI 问题：
【例如入口弱、转化低、按钮误触、状态反馈弱、品牌割裂；没有则写：暂无】
```

## 强制知识库巡视 Hook

当用户提出视觉洞察、UI 视觉分析、竞品 UXUI 分析、基于 PRD 的视觉设计洞察、设计前置 Checklist、UI 方案前设计判断、视觉走查或设计评审前分析时，必须先执行“视觉洞察知识库巡视流程”，再输出最终内容。

不要仅凭通用审美、个人偏好或泛泛设计经验直接给结论。

### 第一步：读取知识库

进行视觉判断前，必须读取：

1. `references/视觉原则索引.md`
2. `references/视觉原则知识库.md`

如果任务包含竞品分析，还必须读取：

3. `references/竞品UXUI分析框架.md`

如果用户需要最终结构化交付物，还必须读取：

4. `references/输出模板.md`

如果需要判断输入材料是否充分，读取：

5. `references/输入与工作流.md`

### 第二步：建立“原则 - 证据”映射

每一次视觉洞察任务，都必须先判断当前需求命中了哪些视觉设计原则，并建立以下映射：

- 业务目标
- 用户角色与使用场景
- 关键路径步骤
- 涉及的 UI 元素
- 命中的设计原则
- 证据来源：PRD / 链路截图 / 当前 UI / 竞品截图 / 历史 UI 问题
- 对用户判断或行为的影响
- 可关联的设计指标

只有当某条设计原则能够对应到具体业务路径、具体 UI 元素或具体用户判断时，才允许使用这条原则。不要为了显得专业而硬套原则。

### 第三步：用原则作为判断标准

进行视觉洞察时，必须基于知识库巡视：

- 第一眼落点是否正确
- 信息层级是否服务业务优先级
- 视觉动线是否能引导用户进入下一步
- 主 CTA 是否清晰、稳定、易识别
- 页面分组和布局是否降低理解成本
- 高频操作是否便捷、易触达
- 状态反馈是否完整
- 加载、成功、失败、空状态是否形成闭环
- 是否提前预防错误和误触
- 是否符合品牌一致性
- 是否能建立信任感和安全感
- 文案、字号、对比度是否具备可读性
- 移动端是否适配单手操作和小屏场景
- 是否修复历史 UI 问题

不要把原则当成理论解释来输出。必须把原则转化为具体 UI 观察、具体问题判断或具体设计动作。

### 第四步：输出有依据的视觉洞察

每一条重要视觉建议都必须包含：

- 问题 / 机会点
- 证据来源
- 对应路径步骤
- 相关 UI 元素
- 建议的 UI 设计动作
- 命中的设计原则
- 预期影响
- 关联设计指标
- 优先级

如果某条建议主要来自业务判断，而不是原则库直接推导，标注：

> 该建议属于业务判断，不属于原则库直接推导。

### 第五步：禁止泛化表达

禁止只写以下模糊表达：

- 提升高级感
- 优化体验
- 强化氛围
- 增强视觉吸引力
- 更加简洁
- 更有品质感
- 更年轻化
- 更专业
- 更清晰
- 更好看

除非同时说明：具体改哪个 UI 元素、为什么改、服务哪个路径步骤、解决什么用户判断问题、对应哪个设计原则、影响哪个指标。

## 工作流

1. 判断输入材料完整度，必要时引导用户补充。
2. 读取视觉原则索引与知识库。
3. 如果包含竞品分析，读取竞品 UXUI 分析框架。
4. 如果要输出完整报告，读取输出模板。
5. 提炼需求视觉目标：业务目标、用户判断、视觉目标、关键指标。
6. 拆解视觉用户故事：具体角色、具体场景、动作层动机、深层动机。
7. 拆解关键视觉路径：用 `→` 串联主路径，并逐步判断视觉任务。
8. 建立原则 - 证据映射。
9. 输出竞品分析或竞品采集清单。
10. 输出当前 UI / 链路视觉洞察、Checklist 与可执行设计建议。

## 输出要求

最终内容必须帮助 UI 设计师回答：

1. 当前需求的视觉设计目标是什么？
2. 用户在每个路径步骤中需要判断什么？
3. 哪些 UI 元素应该突出？
4. 哪些 UI 元素应该弱化？
5. 哪些状态必须补齐？
6. 哪些历史 UI 问题需要修复？
7. 哪些竞品表达值得借鉴或规避？
8. 哪些设计原则支撑这些判断？
9. 这些设计动作影响哪些指标？

如果输出内容无法指导 UI 设计动作，继续补充，直到形成可执行建议。

## 质量底线

- 不编造竞品截图中不存在的细节。
- 不把设计原则当百科解释。
- 不把“视觉风格选择库”纳入分析，除非用户再次明确要求。
- 不默认所有需求都需要营销化视觉。
- 不用审美偏好替代业务路径判断。
- 每条关键建议都要尽量绑定路径步骤、UI 元素、原则和指标。
